# Bot Players + Disconnect Cleanup — Design Spec

Date: 2026-05-18

## Overview

Two related features:

1. **Bot players** — host can add AI-controlled players that occupy real seats, bet using a Kelly-adjacent Hi-Opt II bet ramp, play using Hi-Opt II basic strategy with index deviations, occasionally emote, and leave when broke.
2. **Disconnect cleanup** — host automatically removes stale disconnected player entries so their seats are freed for new players (or bots).

---

## Part 1: Bot Players

### Data Model

Bots are stored as normal player entries in `rooms/${code}/players/${botUid}` with one extra field: `isBot: true`. No separate Firebase path. All existing rendering, dealing, and resolution code treats them as normal players.

**Bot UID format:** `bot_${Date.now()}` — unique, generated by the host, not a Firebase Auth UID. Allowed by the existing player write rule which permits the host to write to any player path.

**Host-local state** (added to `game.js` module scope alongside existing `runningCount`, `localDeck`):
```js
const botUids = new Set();  // which UIDs are bots — host only
let hiOptIICount = 0;       // separate from runningCount (Hi-Lo)
```

On `initHostFeatures()`, the host scans `room.players` for existing `isBot: true` entries and repopulates `botUids`. Partial reconnect recovery — existing "host refresh loses the deck" limitation still applies.

**Bot starting balance:** same value as room `settings.startingBalance` (already set by host in lobby).

**Seat limit:** up to 6 total players (bots + humans). `/addbot` is rejected if the active (non-kicked) seat count is already 6.

---

### New Module: `js/bot.js`

Pure logic, no Firebase, no DOM imports.

#### Hi-Opt II Card Values

```
A=0, 2=+1, 3=+1, 4=+2, 5=+2, 6=+1, 7=+1, 8=0, 9=0, 10/J/Q/K=-2
```

Export: `hiOptIIValue(card)` — returns the integer tag value for a card object.

#### Basic Strategy Table

Standard 6-deck H17 basic strategy encoded as three lookup objects keyed by `"playerValue_dealerUpcard"`:

- **Hard totals** — player hard values 5–17 vs dealer upcards 2–A
- **Soft totals** — soft 13–21 (Ace + X) vs dealer upcards 2–A
- **Pairs** — 2-2 through A-A vs dealer upcards 2–A

Actions: `hit`, `stand`, `double`, `split`, `surrender`. Where double/split isn't available (e.g. > 2 cards), fall back to the next best action (`double` → `hit`, `split` → `hit`).

Export: `basicStrategy(playerHand, dealerUpcard, settings)` — returns action string.

#### True Count Index Deviations

Applied on top of basic strategy. Key plays (Hi-Opt II):

| Situation | Deviation |
|---|---|
| 16 vs 10 | Stand at TC ≥ 0 (basic: hit) |
| 15 vs 10 | Stand at TC ≥ +4 (basic: hit) |
| 12 vs 4 | Stand at TC ≥ 0 (basic: hit) |
| 12 vs 3 | Stand at TC ≥ +2 (basic: hit) |
| 12 vs 2 | Stand at TC ≥ +3 (basic: hit) |
| 9 vs 2 | Double at TC ≥ +1 (basic: hit) |
| 9 vs 7 | Double at TC ≥ +3 (basic: hit) |
| 11 vs A | Double at TC ≥ +1 (basic: hit, H17 game) |
| 10 vs 10 | Double at TC ≥ +4 (basic: stand) |
| 10 vs A | Double at TC ≥ +3 (basic: hit) |
| Insurance | Take at TC ≥ +3 (if insurance enabled in settings) |

Export: `botDecision(playerHand, dealerUpcard, trueCount, settings, balance)` — applies deviations on top of `basicStrategy`, returns final action string.

#### Bet Ramp

Kelly-adjacent 1–12 unit spread, scaled to starting balance:

```
unit = max(minBet, floor(startingBalance / 100))

TC ≤ 0  →  minBet      (always minimum at neutral/negative count)
TC  1   →  2 × unit
TC  2   →  4 × unit
TC  3   →  6 × unit
TC  4   →  8 × unit
TC ≥ 5  →  12 × unit
```

Result clamped to `[minBet, min(maxBet, currentBalance)]`.

Export: `botBet(trueCount, startingBalance, minBet, maxBet, currentBalance)` — returns integer bet amount.

#### Emote Logic

Uses the existing emoji set `['😂', '😬', '💀', '🔥', '👑', '💸']`.

| Outcome | Chance | Emojis |
|---|---|---|

### Game Flow Integration (`game.js`)

#### Betting Phase

In `handleRoomUpdate` when `phase === 'betting'`, the host iterates `botUids`. For each bot with `status === 'waiting'`:

1. Compute `trueCount = hiOptIICount / (localDeck.length / 52)`
2. Get bot's `startingBalance` from `room.settings.startingBalance`
3. Call `botBet(trueCount, startingBalance, settings.minBet, settings.maxBet, bot.balance)`
4. `updatePlayer(botUid, { bet, status: 'ready' })`

This runs on the same tick as the existing "all players ready → advance" check so bots never hold up the round.

#### Hi-Opt II Count Maintenance

Updated in the same two places `runningCount` is updated:

- **`handleDealingPhase`**: after cards are dealt, sum `hiOptIIValue` over all newly dealt cards (player hands + dealer hand)
- **`applyPlayerAction` hit branch**: add `hiOptIIValue(cardFromStr(card))` for the drawn card

On shoe reshuffle: `hiOptIICount = 0` alongside `runningCount = 0`.

#### Playing Phase

In `watchForPlayerAction`, before the existing `player.action` check:

```js
if (botUids.has(turn)) {
  // don't wait for Firebase action write — act directly
  const delay = 600 + Math.random() * 800; // 600–1400ms
  setTimeout(() => applyPlayerAction(turn, decidedAction, currentRoom), delay);
  return;
}
```

The bot decision reads: dealer upcard from `room.dealer.hand[0]`, bot's hand from `player.hands[player.handIndex || 0]`, true count from local `hiOptIICount / (localDeck.length / 52)`.

For splits: bot gets a second hand added (same as human split flow). After the first hand resolves, `watchForPlayerAction` fires again for the next hand and the bot acts again.

#### Resolution — Emotes

When `phase` changes to `'resolution'` (same guard as the existing `lastCatchphrasePhase` check), the host iterates `botUids`:

1. Resolve each bot's outcome from `room.players[botUid]` (same logic as `determineCatchphraseEvent` but for a given UID)
2. Call `getBotEmote(outcome)` — if non-null, `sendEmojiReaction(roomCode, botUid, emoji)` after `500 + Math.random() * 1500` ms

Emotes fire from the bot's player spot via the existing `spawnFloatingEmoji` / `data-uid` system.

#### Broke Check — Round Reset

In the existing round-reset `setTimeout` (where `nextStatus` is decided), after balances are written, the host checks each bot in `botUids`:

```js
if (bot.balance <= 0) {
  await kickPlayer(roomCode, botUid);
  botUids.delete(botUid);
  await sendSystemMessage(roomCode, `${bot.name} is out of chips and left the table.`);
}
```

Kicked bots are filtered from rendering by the existing `!p.kicked` check in `ui.js`.

#### Bots Added Mid-Round

Bots added while `phase !== 'betting'` are written with `status: 'sitting-out'` (same as humans joining mid-round). They participate starting next betting phase.

### Commands

#### `initChat` Signature Change

```js
initChat(roomCode, uid, name, { onAddBot, onRemoveBot } = {})
```

Fourth param is optional — non-host clients pass nothing. Keeps chat.js free of game.js imports.

#### `/addbot` (host-only)

Guards: isHost, `onAddBot` present, active seat count < 6.  
Flow: `onAddBot(room)` → game.js picks name, generates UID, writes Firebase entry, adds to `botUids`, sends system message `"[Name] joined the table."`.

#### `/removebot <name>` (host-only)

Guards: isHost, `onRemoveBot` present, name provided.  
Flow: `onRemoveBot(name, room)` → game.js finds bot by case-insensitive name match in `room.players` where `isBot === true`, calls `kickPlayer`, removes from `botUids`, sends system message `"[Name] was removed from the table."`.  
Error if no matching bot found.

#### `/kick` on bots

Works as-is — `kickPlayer` sets `kicked: true` regardless of `isBot`. Game.js `onRoomChange` checks if kicked UID is in `botUids` and removes it.

## Part 2: Disconnect Cleanup

### Problem

`connected: false` players are excluded from rendering as empty seats but still count against the 6-seat limit in `joinRoom` (`filter(p => !p.kicked)`). A player who closes their browser holds a seat indefinitely.

### Fix

Host removes stale disconnected player entries from Firebase after a grace period.

**In `game.js` (host-only), new module-level map:**
```js
const disconnectTimers = new Map(); // pid → timeoutId
```

**In `handleRoomUpdate`**, the host iterates `room.players`:

- For each player where `connected === false && !kicked`:
  - If not already in `disconnectTimers`: start a 30-second timer
  - When timer fires: if player is still disconnected AND `phase` is not `'playing'` or `'dealing'` → call `kickPlayer(roomCode, pid)` and clear the timer entry (uses existing `kicked: true` mechanism; rendering already filters kicked players out)
  - If `phase` is `'playing'` or `'dealing'`: wait — the existing `watchForPlayerAction` force-stand handles them mid-round; cleanup fires at the next `handleRoomUpdate` call once the phase has moved on
- For each player where `connected === true || kicked`: clear any pending timer for that pid (they reconnected or were already handled)

**Bots:** bots have `connected: true` always and never trigger disconnect cleanup.

**Firebase rule:** `remove()` on a player path is a write — already covered by the host write rule.

## Constraints and Known Limitations

- If the host refreshes mid-game, `botUids` and `hiOptIICount` and `localDeck` are all lost. `botUids` is partially recovered via `isBot: true` scan but the count and deck are gone — existing limitation, not addressed here.
- Basic strategy table covers standard 6-deck H17 rules. If room settings use unusual rules (surrender off, DAS off, etc.) the fallback logic handles unsupported moves gracefully.
- Bot names are a placeholder list — user will supply the final list before implementation.
